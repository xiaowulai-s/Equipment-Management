# 工业设备管理系统 v2.0 - 新架构说明

## 概述

工业设备管理系统 v2.0 基于四层解耦架构重构,采用插件化设计,支持多设备并发管理和实时数据监控。

**核心特性**:
- ✅ 四层解耦架构:协议层 → 通信层 → 设备层 → 数据/UI层
- ✅ 插件化协议设计,易于扩展新协议
- ✅ 完整的数据持久化 (SQLite + Repository 模式)
- ✅ 线程安全的报警系统 (四级报警 + 死区 + 升级/降级)
- ✅ 现代化 UI (Fluent Design 风格,深色主题)
- ✅ 完整的测试覆盖 (1800+ 项测试用例)

---

## 四层解耦架构

### 1. 协议层 (Protocol Layer)

**位置**: `src/protocols/`

**职责**:
- 定义统一的协议抽象接口
- 处理协议帧打包/解包
- 提供协议特定的校验逻辑

**组件**:
- `base_protocol.py` - 协议基类
  - `BaseProtocol` - 抽象基类
  - `ReadResult` / `WriteResult` - 读写结果数据类
  - `ModbusFunctionCode` - 功能码枚举
  - `RegisterType` - 寄存器类型枚举
  - `DataType` - 数据类型枚举 (含大小和位数)
  - `ByteOrder` - 字节序枚举
  - `ProtocolType` - 协议类型枚举

- `modbus_tcp.py` - Modbus TCP 协议
  - 支持 FC 03 (读保持寄存器)
  - 支持 FC 04 (读输入寄存器)
  - 支持 FC 06 (写单个寄存器)
  - 支持 FC 16 (写多个寄存器)
  - MBAP 头处理 (事务ID + 协议ID + 长度 + 单元ID)

- `modbus_rtu.py` - Modbus RTU 协议
  - CRC-16 校验 (多项式 0xA001)
  - 支持所有 TCP 功能码
  - 帧间延时 (3.5 字符时间)

- `modbus_ascii.py` - Modbus ASCII 协议
  - LRC 校验 (纵向冗余校验)
  - ASCII 编解码
  - 回车换行符 (CR/LF) 分隔

**测试覆盖**:
- ModbusTCP: 80 项测试 ✅
- ModbusRTU/ASCII: 117 项测试 ✅

---

### 2. 通信驱动层 (Communication Driver Layer)

**位置**: `src/communication/`

**职责**:
- 提供底层 I/O 抽象
- 处理字节流缓冲区逻辑
- 解决工业现场的粘包、断包问题

**组件**:

- `base_driver.py` - 通信驱动基类
  - `BaseDriver` - 抽象基类
  - `DriverState` - 状态机枚举 (IDLE/CONNECTING/CONNECTED/DISCONNECTING/CLOSED/ERROR)
  - `DriverStats` - 性能统计 (发送/接收字节数, 请求/成功/失败次数)
  - 自动重连机制
  - Keepalive 心跳

- `tcp_driver.py` - TCP 驱动
  - socket 非阻塞 I/O
  - select 超时控制
  - TCP Keepalive 选项
  - SSL/TLS 加密支持 (可选)
  - 自动重连策略

- `serial_driver.py` - 串口驱动
  - 基于 pyserial
  - 硬件流控 (RTS/CTS)
  - 软件流控 (XON/XOFF)
  - 热插拔检测 (Windows/Linux)
  - 设备枚举

**测试覆盖**:
- BaseDriver: 状态机 + 性能监控 + 线程安全 ✅
- TcpDriver: 60+ 项测试 ✅
- SerialDriver: 40+ 项测试 ✅

### 3. 设备管理层 (Device Management Layer)

**位置**: `src/device/`

**职责**:
- 设备工厂:根据协议类型动态创建设备
- 设备管理:多设备生命周期管理
- 数据采集引擎:轮询、写入、统计

**组件**:

- `enums.py` - 枚举定义
  - `DeviceStatus` - 设备状态 (DISCONNECTED/CONNECTING/CONNECTED/ERROR)
  - `PollMode` - 轮询模式 (AUTO/MANUAL)
  - `AccessMode` - 访问模式 (READ_ONLY/READ_WRITE/WRITE_ONLY)

- `models.py` - 领域模型
  - `TcpParams` - TCP 连接参数 (host, port, timeout, keepalive_enabled)
  - `SerialParams` - 串口连接参数 (port, baud_rate, data_bits, stop_bits, parity, flow_control)
  - `PollConfig` - 轮询配置 (interval_ms, retry_count, retry_interval_ms, timeout_ms)
  - `AlarmConfig` - 报警配置 (high_high, high, low, low_low, deadband, enabled, level)
  - `AlarmLevel` - 报警级别 (INFO/WARNING/ERROR/CRITICAL)
  - `Register` - 寄存器模型 (含报警检测 `_check_alarm()`)
  - `Device` - 设备模型 (含设备状态、寄存器字典、配置)

- `device_manager.py` - 设备管理器
  - CRUD 操作 (add_device/remove_device/update_device/get_device)
  - 搜索功能 (search_devices/by_status/by_protocol/by_name)
  - 批量操作 (connect_all/disconnect_all/poll_all/stop_all)
  - 信号聚合 (device_added/device_removed/device_connected/device_disconnected/device_data_updated/device_error)
  - JSON 持久化 (save_to_json/load_from_json)

- `data_collector.py` - 数据采集引擎
  - `CollectorStats` - 线程安全统计 (threading.Lock), 100 次延迟滑动窗口
  - `PollWorker(QThread)` - 单设备轮询线程, 失败重试, 请求合并, 自动连接管理
  - `DataCollector(QObject)` - 主控制器, 全局启停, 单设备控制, 写入路由, 统计聚合
  - `create_protocol()` - 协议工厂 (TCP/RTU/ASCII)
  - `reg_type_is_bit()` - 功能码辅助判断

- `__init__.py` - 包导出

**测试覆盖**:
- 设备模型: 180 项测试 ✅
- 设备管理器: 144 项测试 ✅
- 数据采集引擎: 14 节, ~80 项测试 ✅

### 4. 数据持久化层 (Data Persistence Layer)

**位置**: `src/data/`

**职责**:
- 数据库管理 (SQLite)
- ORM 模型映射
- Repository 模式封装数据访问

**组件**:

- `database.py` - 数据库管理器
  - `DatabaseManager` - 单例/多例模式
  - WAL 模式 (提高并发性能)
  - 外键约束
  - 上下文 session 自动 commit/rollback
  - 初始化表结构

- `models.py` - ORM 模型 (SQLAlchemy 2.0)
  - `DeviceModel` - 设备表
  - `RegisterMapModel` - 寄存器映射表
  - `HistoricalDataModel` - 历史数据表
  - `AlarmRecordModel` - 报警记录表
  - `AlarmRuleModel` - 报警规则表
  - `SystemLogModel` - 系统日志表
  - 支持 `from_domain()` / `to_domain()` 与运行时 Device/Register 双向转换

- `repository/` - Repository 实现
  - `base.py` - `BaseRepository` 泛型基类 (CRUD)
  - `device_repository.py` - `DeviceRepository` (save_device, 搜索, 预加载)
  - `historical_repository.py` - `HistoricalDataRepository` (记录/查询/统计/清理)
  - `alarm_repository.py` - `AlarmRecordRepository` + `AlarmRuleRepository`
  - `__init__.py` - 包导出

**数据库表结构**:

```sql
-- 设备表
CREATE TABLE devices (
    device_id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    protocol_type TEXT NOT NULL,
    device_type TEXT NOT NULL,
    tcp_params JSON,
    serial_params JSON,
    poll_config JSON,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);

-- 寄存器映射表
CREATE TABLE register_maps (
    id INTEGER PRIMARY KEY,
    device_id TEXT NOT NULL,
    name TEXT NOT NULL,
    address INTEGER NOT NULL,
    register_type TEXT NOT NULL,
    data_type TEXT NOT NULL,
    byte_order TEXT,
    access_mode TEXT,
    poll_interval_ms INTEGER,
    alarm_config JSON,
    FOREIGN KEY (device_id) REFERENCES devices(device_id)
);

-- 历史数据表
CREATE TABLE historical_data (
    id INTEGER PRIMARY KEY,
    device_id TEXT NOT NULL,
    register_address INTEGER NOT NULL,
    timestamp TIMESTAMP NOT NULL,
    value REAL NOT NULL,
    quality TEXT,
    FOREIGN KEY (device_id) REFERENCES devices(device_id)
);

-- 报警记录表
CREATE TABLE alarm_records (
    id INTEGER PRIMARY KEY,
    device_id TEXT NOT NULL,
    register_address INTEGER,
    level TEXT NOT NULL,
    message TEXT NOT NULL,
    triggered_at TIMESTAMP NOT NULL,
    cleared_at TIMESTAMP,
    acknowledged_at TIMESTAMP,
    acknowledged_by TEXT,
    FOREIGN KEY (device_id) REFERENCES devices(device_id)
);

-- 报警规则表
CREATE TABLE alarm_rules (
    id INTEGER PRIMARY KEY,
    device_id TEXT NOT NULL,
    register_address INTEGER,
    rule_type TEXT NOT NULL,
    threshold REAL,
    enabled BOOLEAN,
    FOREIGN KEY (device_id) REFERENCES devices(device_id)
);

-- 系统日志表
CREATE TABLE system_logs (
    id INTEGER PRIMARY KEY,
    timestamp TIMESTAMP NOT NULL,
    level TEXT NOT NULL,
    logger TEXT,
    message TEXT NOT NULL,
    extra_data JSON
);
```

**注意**:
- DataType/ProtocolType 枚举的 .value 可能是 tuple (name, size, bits), 存储时取 [0]
- 所有 Repository 操作仅 flush 不 commit, 事务控制交给 session 上下文

### 5. 报警系统 (Alarm System)

**位置**: `src/alarm/`

**职责**:
- 报警信号聚合
- 报警记录持久化
- 报警确认和统计
- 通知分发

**组件**:

- `alarm_manager.py` - 报警管理器
  - `AlarmManager(QObject)` - 主控制器
  - `connect_device(device)` - 自动连接报警信号
  - `record_alarm()` / `record_alarm_clear()` - 持久化到 alarm_records 表
  - `acknowledge_alarm()` / `acknowledge_all_by_device()` - 报警确认
  - `get_active_alarms()` / `get_alarm_records()` / `get_statistics()` - 查询
  - `create_rule()` / `delete_rule()` - 报警规则管理
  - 支持无 DB 纯内存模式

- `notification.py` - 通知服务
  - `AlarmNotificationService(QObject)` - 通知渠道分发
  - 渠道: POPUP/SOUND/EMAIL/CUSTOM
  - 冷却机制: 同一设备 + 寄存器在 cooldown_seconds 内不重复通知
  - 级别过滤: min_level 控制最低通知级别
  - 非阻塞弹窗 (QMessageBox.show() 不 exec())

- `__init__.py` - 包导出

**关键设计**:
- 报警阈值检测由 `Register._check_alarm()` 完成 (四级 + 死区 + 升级/降级)
- AlarmManager 仅做信号聚合、记录持久化、确认、统计、通知分发

**测试覆盖**:
- 内存模式: 记录/清除/统计 全部通过 ✅
- 数据库模式: 记录/查询/确认/批量确认/规则 CRUD/清理 全部通过 ✅
- 通知服务: 冷却抑制/级别过滤 全部通过 ✅

## 设计原则

### 高内聚、低耦合
- 每层职责单一明确
- 层间通过信号与槽通信
- 易于扩展和维护

### 插件化
- 协议插件易于添加新协议
- 设备工厂支持动态创建
- 配置驱动,无需硬编码

### 线程安全
- 所有 I/O 操作在 QThread 中执行
- 统计类使用 threading.Lock 保护
- 数据库操作使用 session 上下文管理事务

### 工业级
- 内置重试和错误处理
- 配置持久化和自动恢复
- 完整的日志和状态监控
- 四级报警 + 死区 + 升级/降级

## 快速开始

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 运行程序

```bash
python main.py
```

### 3. 添加设备

1. 点击左侧 "+" 按钮
2. 输入设备名称
3. 选择协议类型 (Modbus TCP/RTU/ASCII)
4. 配置连接参数 (TCP: host/port 或 Serial: port/baudrate)
5. 点击 "确定"

### 4. 连接设备

1. 在设备列表中选择设备
2. 点击 "连接" 按钮
3. 查看右侧实时数据卡片
