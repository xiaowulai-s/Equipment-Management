# 工业设备管理系统 v2.0 — 未来功能规划与可行性分析

> 基于当前已实现的四层解耦架构（协议层→通信驱动层→设备管理层→数据持久化层 + UI层），从六大维度提出未来功能补充建议。

---

## 一、基础设施补强（优先级 P0）

> 这类功能是系统正式投入使用前必须补齐的关键短板。

### 1.1 采集数据自动持久化

**现状**：`PollWorker` 每轮轮询后仅更新内存中的 `Register` 模型，历史数据（`historical_data` 表）从未被写入，`HistoricalDataRepository.batch_record()` 已实现但无人调用。

**规划**：在 `PollWorker._dispatch_values()` 成功回写寄存器值后，将采集到的数据点暂存到线程本地缓冲区；由 `DataCollector` 维护一个定时刷新线程（如每 5 秒），调用 `HistoricalDataRepository.batch_record()` 批量写入数据库。

**作用**：
- 让历史数据查询、趋势分析、数据导出等下游功能有真实数据支撑
- 系统重启后可回溯历史

**可行性**：**高**。`HistoricalDataRepository.batch_record()` 已实现，只需在 `DataCollector` 中新增一个缓冲队列和定时写入线程。改动量约 100-150 行代码，涉及 `data_collector.py` 一个文件。估计 2-3 小时。

**实施难度**：★☆☆☆☆

---

### 1.2 DriverManager 驱动管理器

**现状**：`src/communication/__init__.py` 中已标注 "待步骤3.4实现"，缺少统一的驱动连接池和心跳管理。

**规划**：
- 实现 `DriverManager(QObject)` 管理所有 `TcpDriver`/`SerialDriver` 实例
- 连接池复用：同一 host:port 共享一个 `TcpDriver`
- 心跳检测：定期发送 Modbus FC43 (Read Device ID) 或自定义心跳包，检测设备存活
- 自动重连：连接断开后按 `TcpParams.reconnect_interval` 自动重连
- 连接生命周期管理：idle 超时自动断开

**作用**：
- 避免重复创建连接（当前每设备一个协议实例、一个 socket）
- 提升资源利用效率，减少连接泄漏风险
- 心跳机制让系统对设备掉线能快速感知

**可行性**：**高**。`BaseDriver` 已有完整的信号体系和状态机，`TcpDriver` 已支持 Keepalive。`DriverManager` 本质上是一个 QObject 字典 + 定时器。改动量约 300-400 行，新增 `src/communication/driver_manager.py`。估计 4-6 小时。

**实施难度**：★★☆☆☆

### 1.3 旧架构完全迁移

**现状**：仍有 3 个对话框依赖旧架构 `core.*`：
- `AddDeviceDialog` → 依赖 `core.device.device_factory.DeviceFactory`、`DeviceTypeManager`
- `BatchOperationsDialog` → 依赖 `core.device.device_manager.DeviceManager`
- `DeviceTypeDialog` → 依赖 `core.device.device_type_manager.DeviceTypeManager`
- `DataExportDialog` → 依赖 `core.utils.data_exporter.DataExporter`、`core.utils.logger_v2`

**规划**：逐一将这些对话框的 `core.*` 导入替换为 `src.*` 等价实现。设备工厂用 `Device.from_dict()` 替代，数据导出直接使用 Repository 查询。

**作用**：
- 消除对旧代码的依赖，允许安全删除 `core/` 目录
- 统一代码风格和异常处理规范

**可行性**：**高**。新架构已提供完整的等价 API。每个对话框约 1-2 小时迁移，总计约 6-10 小时。

**实施难度**：★★☆☆☆

## 二、协议与通信扩展（优先级 P1）

> 提升系统的设备兼容性和通信能力。

### 2.1 Modbus UDP 协议

**现状**：仅支持 TCP、RTU、ASCII 三种变体，不支持 Modbus UDP（MBAP 头 + UDP 传输）。

**规划**：新增 `ModbusUDPProtocol` 继承 `BaseProtocol`，使用 Python `socket` 模块的 `SOCK_DGRAM`。帧结构与 TCP 相同（MBAP 头），但无需连接/断开。

**作用**：
- 支持基于 UDP 的 Modbus 网关和设备
- 适用于对实时性要求高但允许丢包的场景

**可行性**：**高**。可复用 `ModbusTCPProtocol` 的编解码逻辑（约 70% 代码共享），只需将 socket 改为 UDP 模式。改动量约 200 行。估计 3-4 小时。

**实施难度**：★☆☆☆☆

---

### 2.2 功能码扩展 (FC22/FC23/FC43)

**现状**：支持 FC01-FC06, FC15, FC16 共 8 个标准功能码，缺少 FC22（屏蔽写寄存器）、FC23（读写多寄存器）、FC43（读设备标识）。

**规划**：
- FC23 (Read/Write Multiple Registers)：一次原子操作完成读写，减少通信次数
- FC22 (Mask Write Register)：按位修改单个寄存器，适用于控制位操作
- FC43 (Read Device Identification)：读取设备厂商/型号/固件版本信息

**作用**：
- FC23 可显著提升批量读写场景的通信效率
- FC43 可自动识别设备信息，用于设备发现和管理

**可行性**：**高**。编解码逻辑在 `BaseProtocol` 中已有完整模板，每个功能码约 50-80 行实现。总计约 200 行，估计 3-5 小时。

**实施难度**：★★☆☆☆

---

### 2.3 非 Modbus 协议支持 (OPC UA / MQTT)

**现状**：仅支持 Modbus 协议族，无法对接 OPC UA 服务器或 MQTT Broker。

**规划（OPC UA）**：
- 新增 `src/protocols/opcua_protocol.py`
- 使用 `asyncua` 库（Python OPC UA 客户端）
- 在 `BaseProtocol` 基础上适配 OPC UA 的 Node 读写语义

**规划（MQTT）**：
- 新增 `src/protocols/mqtt_protocol.py`
- 使用 `paho-mqtt` 库
- 订阅设备主题，解析 JSON/二进制 payload 映射到 Register

**作用**：
- 接入西门子/罗克韦尔等主流 PLC 的 OPC UA 接口
- 支持 IoT 设备通过 MQTT 上报数据
- 大幅扩展系统的设备兼容范围

**可行性**：**中**。协议语义差异较大，需要设计适配层。OPC UA 需要额外的异步框架集成（asyncio ↔ QThread 桥接），MQTT 相对简单但需要定义消息格式规范。每个协议约 500-800 行 + 测试。估计各需 15-20 小时。

**实施难度**：★★★★☆（OPC UA）/ ★★★☆☆（MQTT）

---

### 2.4 通信数据帧捕获与协议分析

**现状**：无通信层数据捕获工具，调试时无法查看实际发送/接收的原始字节。

**规划**：
- 在 `BaseProtocol` 中新增 `frame_captured Signal(bytes, str, bool)` （数据帧、方向「发送/接收」、成功/失败）
- 新增 `CommunicationSniffer` 工具类，记录所有帧到环形缓冲区（可配容量）
- 新增 UI 页面：协议分析器（十六进制视图 + 协议解析 + 时间戳 + 过滤搜索）

**作用**：
- 强大的调试工具，现场排障必备
- 可导出为 Wireshark 兼容格式
- 帮助用户理解 Modbus 协议帧结构

**可行性**：**高**。信号机制已具备，核心工作在 UI 的十六进制视图组件。改动量约 300 行（后端）+ 500 行（UI）。估计 8-10 小时。

**实施难度**：★★★☆☆

## 三、数据能力增强（优先级 P1）

> 让数据"活起来"，从纯监控扩展到数据分析。

### 3.1 历史数据趋势分析页面

**现状**：`TrendChart` 和 `RealTimeChart` 组件已实现，但没有独立的历史数据分析页面。`DataExportDialog` 可以导出 CSV，但无法在 UI 中直接查看历史趋势。

**规划**：
- 新增 `HistoryPage` 页面（在 QStackedWidget 中添加第三页）
- 设备选择器 + 寄存器多选 + 时间范围选择器（日期选择器）
- 使用 `pyqtgraph` 绘制历史曲线（复用 `RealTimeChart` 的渲染逻辑）
- 支持缩放、平移、数据点悬停提示
- 支持叠加显示报警阈值线（HH/H/L/LL）
- 支持数据导出（当前视图 → CSV/图片）

**作用**：
- 操作员可回溯设备历史状态，分析故障前兆
- 叠加报警线帮助判断设备是否接近临界值
- 替代导出 CSV → Excel 的低效流程

**可行性**：**高**。`pyqtgraph` 组件和 Repository 查询已就绪，主要工作是 UI 布局和交互逻辑。约 600-800 行代码。估计 10-15 小时。

**实施难度**：★★★☆☆

---

### 3.2 报警看板（Alarm Dashboard）

**现状**：报警功能（`AlarmManager` + `AlarmNotificationService`）已实现，但主窗口中 `_show_alarm_history()` 仅用一个 QMessageBox 简单弹窗展示。缺少独立的报警浏览/管理界面。

**规划**：
- 新增 `AlarmPage` 页面
- **实时报警列表**：显示当前活跃报警（表格：设备名/寄存器/值/级别/时间/确认状态），支持确认操作
- **报警历史查询**：时间范围 + 设备筛选 + 级别过滤 + 已确认/未确认过滤
- **报警统计图表**：按小时/天/周的报警次数柱状图（pyqtgraph BarGraphItem）
- **报警闪烁提示**：新报警到达时列表行闪烁 + 状态栏计数更新

**作用**：
- 运维人员的核心工作台
- 报警统计帮助识别频发故障设备和时间段

**可行性**：**高**。`AlarmRecordRepository` 已提供 `get_active()`、`get_unacknowledged()`、`get_statistics()` 等查询方法。核心工作量在 UI。约 800-1000 行。估计 12-18 小时。

**实施难度**：★★★☆☆

---

### 3.3 系统总览仪表盘（Dashboard）

**现状**：欢迎页仅显示"欢迎使用"静态文本，没有数据概览。

**规划**：
- 改造欢迎页为 `DashboardPage`
- **关键指标卡片**：设备总数/在线数/离线数/报警数/采集成功率/平均延迟
- **设备状态饼图**：按状态分布（已连接/未连接/连接中/错误）
- **协议分布柱状图**：TCP/RTU/ASCII 设备数量
- **最近报警列表**（Top 10）
- **最近通信异常**（Top 10）
- **系统资源监控**：CPU/内存/数据库大小（使用 `psutil`）

**作用**：
- 管理层一目了然掌握系统整体健康状况
- 快速定位问题设备

**可行性**：**高**。`DeviceManager.get_statistics()` 已返回完整统计数据，`AlarmManager` 提供活跃报警查询。UI 工作使用现有 `DataCard` 和 `pyqtgraph` 组件即可。约 600-800 行。估计 10-15 小时。

**实施难度**：★★★☆☆

---

### 3.4 数据备份与恢复

**现状**：无数据库备份/恢复功能。数据丢失风险较高。

**规划**：
- 定时自动备份：每日/每周自动复制 SQLite 数据库文件到 `backups/` 目录
- 手动备份：菜单 → "数据库备份"，选择保存路径
- 备份管理：保留最近 N 份备份，自动清理过期备份
- 恢复功能：选择备份文件 → 确认 → 替换当前数据库 → 重启应用
- 导入向导：支持从备份文件恢复设备配置（不覆盖运行时状态）

**作用**：
- 数据安全保障
- 系统迁移/升级时的数据迁移手段

**可行性**：**高**。SQLite 单文件数据库，备份就是文件复制。实现简单，约 200 行 + UI 对话框。估计 4-6 小时。

**实施难度**：★☆☆☆☆

## 四、UI/UX 增强（优先级 P2）

> 提升用户体验和界面专业度。

### 4.1 国际化 (i18n) 支持

**现状**：所有 UI 文本硬编码中文（`TextConstants`、`UIMessages`、各对话框），无多语言支持。

**规划**：
- 使用 PySide6 内置的 `QTranslator` + `.ts` 翻译文件
- 新增 `ui/i18n/` 目录，初始支持中文（zh_CN）和英文（en_US）
- 提取所有硬编码文本到翻译资源文件
- 设置对话框中新增"语言"选项

**作用**：
- 面向国际市场的必要条件
- 便于开源社区贡献翻译

**可行性**：**中**。PySide6 的 `lupdate` + `linguist` 工具链成熟，但文本提取工作量较大（约 500+ 处字符串）。翻译文件创建约 2-3 小时，文本提取和替换约 15-20 小时。估计总计 20 小时。

**实施难度**：★★★☆☆

---

### 4.2 主题编辑器

**现状**：仅支持内置深色/浅色主题切换，无法自定义颜色和样式。

**规划**：
- 新增 `ThemeEditorDialog` 对话框
- 可视化编辑：主色/辅助色/背景色/文字色/强调色（颜色选择器）
- 实时预览：修改后立即反映到主窗口
- 导出/导入主题配置（JSON 文件）
- 预设主题库：提供 5-6 个预设主题（GitHub Dark、Solarized、Monokai、Nord 等）

**作用**：
- 企业可根据品牌色定制系统外观
- 深色模式用户可根据偏好微调

**可行性**：**中**。`ThemeManager` 已有主题切换框架，扩展为动态 QSS 生成即可。颜色选择器可用 `QColorDialog`。约 400-600 行。估计 8-12 小时。

**实施难度**：★★★☆☆

### 4.3 设备分组树形导航

**现状**：设备列表使用扁平 `QTreeWidget`，所有设备在同一层级显示。`Device.group_name` 字段已存在但未在 UI 中利用。

**规划**：
- 改造左侧设备树为分组树结构
- 顶级节点：协议分组（TCP / RTU / ASCII）或自定义分组（按 `group_name`）
- 二级节点：设备（显示名称 + 状态指示灯）
- 支持拖拽移动设备到不同分组
- 支持右键菜单：新建分组/重命名/删除分组
- 搜索框支持按分组过滤

**作用**：
- 大量设备时组织更有序
- 快速定位特定协议/区域的设备

**可行性**：**高**。`QTreeWidget` 原生支持多级节点和拖拽。数据源 `DeviceManager` 已有 `get_devices_by_protocol()` 和分组查询。约 300-400 行改动。估计 6-8 小时。

**实施难度**：★★☆☆☆

### 4.4 设备模板与克隆

**现状**：每次添加设备需要手动配置所有参数和寄存器。同类设备（如多台相同型号 PLC）配置重复劳动。

**规划**：
- 设备模板：将设备配置（通信参数 + 寄存器列表）保存为模板（JSON 文件，存储在 `config/templates/`）
- 克隆设备：右键设备 → "克隆"，自动填充相同配置，仅修改名称和 IP
- 模板管理对话框：查看/编辑/删除/导入/导出模板
- 内置常用设备模板（Modbus TCP 通用、西门子 S7-200、施耐德 M340 等）

**作用**：
- 大幅减少重复配置工作
- 新项目快速部署

**可行性**：**高**。`Device` 已有完整的 JSON 序列化/反序列化，`Device.from_dict()` 支持从字典创建。模板就是去掉运行时状态的 Device JSON。约 300 行后端 + 400 行 UI。估计 8-10 小时。

**实施难度**：★★☆☆☆

---

### 4.5 寄存器批量导入 (Excel/CSV)

**现状**：寄存器配置只能通过 `RegisterConfigDialog` 手动逐行添加。

**规划**：
- 支持从 Excel (.xlsx) / CSV 文件批量导入寄存器配置
- 格式规范：地址 | 名称 | 类型 | 数据类型 | 缩放系数 | 单位 | 报警阈值
- 导入预览：显示解析结果，用户确认后才写入
- 错误处理：格式错误的行标红，跳过不中止
- 导出模板：生成空模板文件供用户填写

**作用**：
- 设备有上百个寄存器时，手动添加不现实
- 从设备厂商提供的寄存器表直接导入

**可行性**：**高**。使用 `openpyxl`（Excel）和 Python 标准库 `csv`。解析逻辑简单，约 200 行。UI 预览表格约 300 行。估计 5-7 小时。

**实施难度**：★★☆☆☆

## 五、运维与安全（优先级 P2）

> 提升系统在生产环境中的可靠性和安全性。

### 5.1 用户系统与权限控制

**现状**：无用户表、无登录、无权限控制。任何人打开程序即拥有全部权限。

**规划**：
- 新增 `users` 表：用户名、密码哈希（bcrypt）、角色（管理员/操作员/只读）、创建时间
- 登录对话框：启动时显示，验证后进入主界面
- 角色权限矩阵：

| 操作 | 管理员 | 操作员 | 只读 |
|------|--------|--------|------|
| 查看设备数据 | ✅ | ✅ | ✅ |
| 确认报警 | ✅ | ✅ | ❌ |
| 添加/编辑/删除设备 | ✅ | ❌ | ❌ |
| 写入寄存器 | ✅ | ❌ | ❌ |
| 系统设置 | ✅ | ❌ | ❌ |
| 用户管理 | ✅ | ❌ | ❌ |

- 会话管理：登录超时自动退出（可配）
- 操作审计日志：关键操作记录到 `system_logs` 表

**作用**：
- 满足工业环境安全合规要求
- 防止误操作（如误写寄存器导致设备异常）

**可行性**：**中**。需要新增数据表、登录 UI、权限检查中间件。`bcrypt` 库处理密码哈希。约 500 行后端 + 300 行 UI。估计 15-20 小时。

**实施难度**：★★★★☆

---

### 5.2 报警通知渠道完善

**现状**：`AlarmNotificationService` 支持 POPUP/SOUND/EMAIL/CUSTOM，但 EMAIL 渠道为占位实现（未完成 SMTP 配置）。

**规划**：
- **完善 EMAIL**：实现完整的 SMTP 配置（服务器/端口/SSL/用户名/密码），支持 HTML 格式邮件
- **新增 Webhook**：通用 HTTP POST 通知，支持企业微信/钉钉/飞书的自定义机器人
- **新增短信（SMS）**：通过阿里云/腾讯云 SMS API 发送短信通知（可选依赖）
- **报警升级机制**：报警未确认 N 分钟后自动升级通知级别（如：5分钟未确认 → 通知主管）

**作用**：
- 关键报警及时送达运维人员
- 多渠道冗余确保通知不丢失
- 报警升级避免无人处理

**可行性**：**高**（EMAIL + Webhook）/ **中**（SMS）。SMTP 发送是标准库能力，Webhook 就是 `requests.post()`。短信需要第三方 SDK。每渠道约 100-200 行。估计 6-10 小时。

**实施难度**：★★☆☆☆

### 5.3 数据库版本迁移框架

**现状**：数据库变更通过手动执行 `migrate_database.py`，无自动化版本管理。

**规划**：
- 引入 Alembic（SQLAlchemy 配套迁移工具）
- 每次数据库 schema 变更生成迁移脚本
- 应用启动时自动检测并执行待执行的迁移
- 支持升级和降级（回滚）

**作用**：
- 版本升级时数据库变更自动化
- 避免手动迁移导致的数据丢失

**可行性**：**高**。项目已使用 SQLAlchemy ORM，Alembic 是其官方迁移工具，集成简单。约 2-3 小时配置 + 后续每次变更生成迁移脚本。估计初次引入 4-6 小时。

**实施难度**：★★☆☆☆

### 5.4 自动更新机制

**现状**：用户需手动下载新版本替换程序文件。

**规划**：

- 支持增量下载（仅下载变更文件）

**作用**：
- 降低版本管理成本
- 确保用户使用最新版本（安全修复、功能改进）

**可行性**：**中**。需要搭建更新服务（或利用 GitHub Releases API），设计下载和替换流程（程序不能替换自身，需要外部更新器）。约 500 行。估计 10-15 小时。

**实施难度**：★★★★☆

## 六、架构演进（优先级 P3）

> 长期架构改进方向。

### 6.1 插件系统

**现状**：功能全部硬编码在主程序中，扩展需要修改源码。

**规划**：
- 定义插件接口（`PluginProtocol` ABC）：`on_load()`, `on_unload()`, `get_menu_items()`, `get_device_widgets()`
- 插件发现：扫描 `plugins/` 目录，动态加载 Python 模块
- 插件沙箱：每个插件运行在独立命名空间，不能修改核心数据结构
- 插件市场：提供官方插件（OPC UA 适配器、报表生成器、数据转发器等）

**作用**：
- 第三方可开发自定义功能扩展
- 核心代码保持精简稳定

**可行性**：**中**。Python 动态导入和 ABC 接口设计简单，但插件隔离和生命周期管理需要仔细设计。约 800-1000 行框架代码。估计 20-30 小时。

**实施难度**：★★★★☆

---

### 6.2 Web 远程监控

**现状**：系统仅支持本地桌面客户端访问。

**规划**：
- 方案 A：嵌入式 Web 服务器（Flask/FastAPI + WebSocket）
  - 提供 REST API 查询设备状态、历史数据、报警记录
  - WebSocket 实时推送设备数据更新
  - 简单 Web 仪表盘（HTML + Chart.js / ECharts）
- 方案 B：QWebEngineView 嵌入浏览器
  - 在桌面应用中嵌入 Web 页面
  - 通过 QWebChannel 桥接 Python ↔ JavaScript

**作用**：
- 运维人员通过浏览器远程查看设备状态
- 移动端/平板访问（响应式 Web UI）
- 与其他系统集成（数据中台、MES、ERP）

**可行性**：**中**（方案 A）/ **高**（方案 B）。方案 A 需要额外的 Web 框架和前端开发。方案 B 可复用现有 pyqtgraph 组件。方案 A 约 30-40 小时，方案 B 约 15-20 小时。

**实施难度**：★★★★★（方案 A）/ ★★★☆☆（方案 B）

---

### 6.3 数据转发 (Edge → Cloud)

**现状**：数据仅存储在本地 SQLite，无法上传到远程服务器或云平台。

**规划**：
- 新增 `DataForwarder` 服务（QThread）
- 支持协议：MQTT、HTTP POST、WebSocket
- 数据格式：JSON（设备ID/寄存器名/值/时间戳/质量码）
- 转发策略：全量转发 / 变化触发 / 压缩聚合
- 离线缓存：网络断开时本地缓存，恢复后自动补发
- 支持对接：InfluxDB、TimescaleDB、阿里云 IoT Hub、自建平台

**作用**：
- 融入企业数据中台和大数据分析体系
- 为 AI 预测性维护提供数据基础

**可行性**：**中**。MQTT/HTTP 客户端实现简单，离线缓存需要设计队列和确认机制。约 600-800 行。估计 15-20 小时。

**实施难度**：★★★★☆

---

### 6.4 预测性维护 (AI/ML)

**现状**：纯监控+报警模式，无法预测设备故障。

**规划**：
- 基于历史数据的趋势分析（线性回归/指数平滑预测未来值）
- 异常检测：基于统计方法（3σ 规则）自动识别数据异常模式
- 设备健康评分：综合通信成功率、报警频率、数据漂移等指标计算健康度
- 维护建议：根据趋势预测生成维护提醒

**作用**：
- 从"事后报警"升级为"事前预防"
- 延长设备使用寿命
- 降低非计划停机损失

**可行性**：**低**。需要成熟的机器学习管道和充足的训练数据。基础统计方法（移动平均、3σ）可先实现，但真正的预测性维护需要 scikit-learn/PyTorch 集成和专业领域知识。估计 40-60 小时（基础版）。

**实施难度**：★★★★★
